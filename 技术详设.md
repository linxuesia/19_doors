# 19分贝门窗微信小程序后台系统技术详细设计文档

## 1. 系统架构概览

基于您的技术栈偏好（全栈偏前端，Node.js 主导，多端框架，大量图片处理），系统整体架构如下：

- **前端架构：** Taro + React + TailwindCSS。Taro 支持一键编译到微信小程序、H5 等多端，完美契合未来扩展需求，且对 React 开发者非常友好。
- **后端架构：** Node.js + NestJS (企业级/推荐) 或 Express (快速迭代)。NestJS 具有优秀的依赖注入和模块化架构，配合 TypeScript 能提供极强的工程化能力，非常适合这种有复杂 RBAC 权限（多端角色）的系统。
- **数据库：** PostgreSQL 或 MySQL + Prisma (ORM)。Prisma 在 Node.js 生态中具有极其出色的类型推导，提升全栈开发效率。
- **缓存引擎：** Redis。用于存储微信 Session Key、验证码、接口防刷控制、热点数据等。
- **存储方案 (关键)：** 阿里云 OSS / 腾讯云 COS。针对“海量图片”需求，这是重中之重。采用直传+CDN+动调策略。

## 2. 数据库设计 (核心实体ER)

根据最新的原型全貌，系统涉及到门店基本信息、门店入驻、店长入驻、量尺预约、订单流转与案例产品展示。以下是完整的核心数据表设计：

### 2.1 账户与组织实体

**1. User (用户基础表)**
- `id`: UUID 主键
- `wx_openid`: 微信 openid (认证标识)
- `phone`: 手机号 (核心业务标识符)
- `name`: 姓名 / 昵称
- `avatar_url`: 头像
- `role`: 枚举 `CLIENT`(C端客户) | `STORE_OWNER`(老板) | `STORE_MANAGER`(店长) | `INSTALLER`(工人) | `ADMIN`(总台管理)
- `store_id`: 归属门店标识 (针对B端角色1对1绑定，如果是兼职多个门店的工人，后续可扩展 User_Store_Relation 中间表)

**2. Store (门店详情表)**
- `id`: UUID 主键
- `owner_id`: 门店负责人/老板 ID (外键关联 User，确认门店归属权)
- `name`: 门店名称 (如：上海松江直营店)
- `type`: 枚举 `DIRECT`(直营体验店) | `FRANCHISE`(加盟店)
- `cover_image`: 门店封面图 (C端展示用)
- `description`: 门店简介说明
- `business_hours`: 营业时间 (如：09:00 - 18:00)
- `address`: 详细地址
- `latitude` / `longitude`: 经纬度 (支撑 C 端门店查找与距离计算)
- `phone`: 门店联系电话
- `status`: 枚举 `OPEN`, `CLOSED`, `PENDING_REVIEW` (待审核)
- `created_at`: 时间戳

### 2.2 申请与审批流实体

**3. Store_Application (门店入驻申请表)**
用于处理加盟商申请开店的流程，总台 Admin 审批。
- `id`: UUID 主键
- `applicant_id`: 申请人 (对应 User)
- `company_name`: 营业执照名称
- `contact_name`: 联系人姓名
- `phone`: 联系电话
- `business_license_url`: 营业执照照片 (OSS链接)
- `status`: 枚举 `PENDING` (待总台审核) | `APPROVED` (已通过) | `REJECTED` (已驳回)
- `result_store_id`: 若通过，关联新建的 Store ID
- `admin_remarks`: 管理员审批意见
- `created_at`: 时间戳

**4. Staff_Application (员工入驻/绑定申请表)**
用于处理店长、工人加入某个特定门店的申请，由门店 Owner 审批。
- `id`: UUID 主键
- `applicant_id`: 申请人 (微信扫码发起申请)
- `apply_role`: 申请的角色 (`STORE_MANAGER` 或 `INSTALLER`)
- `store_id`: 目标门店 ID
- `status`: 枚举 `PENDING` | `APPROVED` | `REJECTED`
- `reviewer_id`: 审批人 ID (该门店的老板)
- `created_at`: 时间戳

### 2.3 业务流转实体 (核心)

**5. Measurement_Reservation (量尺预约表)**
用于C端客户在线发起预约，或门店帮客户产生的初步线索。

- `client_id`: 客户 ID (关联 User，如已登录)

- `address`: 房屋详细地址 (省市区+门牌号)
- `house_area`: 房屋面积 (可选)
- `expected_date`: 期望量尺日期
- `remarks`: 客户备注留言
- `status`: 枚举 `PENDING`(待处理) | `ASSIGNED`(已分配测量人员) | `MEASURED`(已量尺) | `CANCELED`(已取消)
- `assignee_id`: 被指派去量尺的员工 ID (店长或工长)
- `created_at`: 时间戳

**6. Order (业务订单表)**
从线下开单后录入系统，标志着进入施工和质保交付周期。
- `id`: UUID 主键
- `order_no`: 订单号（业务生成，例如 SO-20231015-001）
- `client_id`: 客户 ID (关联 User)
- `store_id`: 归属门店 ID
- `installer_id`: 安装工 ID (接单人为主负责人，如果有多人施工可存 JSON 数组或扩展关联表)
- `product_name`: 产品信息 (如：S100内开窗)
- `community_name`: 施工小区名称
- `install_address`: 施工详细地址
- `scheduled_install_date`: 预计安装日期
- `blueprint_url`: 订单图纸 (OSS链接)
- `contract_url`: 施工合同电子版备份 (OSS链接)
- `total_amount`: 订单总金额
- `paid_amount`: 已付定金/金额 (支持分期收款进度 tracking)
- `warranty_years`: 质保年限 (1年, 5年, 10年, 终身等)
- `status`: 枚举 `PENDING`(待分配工单), `INSTALLING`(施工中), `REVIEWING`(待完工确认/客户签收), `COMPLETED`(已完工生效)

- `created_at` / `updated_at`

**7. Construction_Log (施工日志表 / 工地动态)**
与订单一对多关联，记录整个施工和流转的过程。
- `id`: UUID 主键
- `order_id`: 归属订单 ID
- `store_id`: 归属门店 ID (数据冗余设计，极大提升按门店查询日志时的性能，方便做数据隔离)
- `stage`: 阶段 (例如：材料入场，拆旧，打胶等)
- `content`: 现场描述信息
- `images`: JSON 数组 (存储 OSS URL)
- `location_info`: 打卡位置信息 (可扩展为 JSON 存储经纬度和定位名称，防工人虚假上传)
- `uploader_id`: 上传人 (工人或店长 ID)
- `is_visible_to_client`: 是否对客户可见 (有时需要审核后再外发，默认 true)
- `created_at`: 时间戳

**8. Showcase_Case (客户案例库表)**
门店将其完工优秀的订单展示到 C 端的“案例”模块。
- `id`: UUID 主键

- `title`: 案例标题 (如：上海汤臣一品 150平系统窗)

- `house_area`: 房屋面积 (如：120平)
- `house_type`: 房型 (如：三室两厅)

- `installer_id`: 施工队长 (关联 User，C端展示服务团队用)

- `category`: 分类 (如：平开窗，推拉门，阳光房)

## 3. 关键技术难题与解决方案

### 3.1 海量图片的存储与传输策略
这是本项目的核心挑战。工人和门店在现场会拍摄大量原始尺寸的图片，如果不加约束会导致极差的用户体验和极高的服务器成本。

- **STS 直接上传模式：** 小程序端**绝对不要**把图片先传给 Node.js 服务器再转存。应由前端向 Node 申请 OSS 临时凭证(STS)，前端使用 `wx.uploadFile` **直传**到 OSS 存储桶。后端零对象流压力。
- **强制压缩上传：** 调起相机/相册时，务必结合小程序的 `wx.compressImage` 在手机本地先做一次画质压缩后再上传。
- **OSS 动态样式 / WebP：** 数据库中只保存图片的云端 Object Key。前端拉取图片展示时，统一使用 CDN 域名拼上动态裁剪参数。比如列表页请求：`image.jpg?x-oss-process=image/resize,w_300/format,webp`。极大降低流量费用，提升加载速度。

### 3.2 角色化鉴权与接口安全 (RBAC)
系统有至少 4 种业务角色，安全隔离至关重要。
- **统一 JWT 网关：** 用户登录后下发 JWT Token，Payload 中携带 `userId` 和 `role`。
- **装饰器鉴权 (NestJS 做法)：**
  ```typescript
  @UseGuards(RolesGuard)
  @Roles(Role.STORE_OWNER, Role.STORE_MANAGER) // 仅老板/店长可访问
  @Post('offline-order')
  async createOfflineOrder() { ... }
  ```
- **数据级隔离：** 对于店长/老板的接口，查询条件必须强制插入 `WHERE store_id = 用户JWT里的store_id`；查询工单时强制 `WHERE installer_id = 当前工人id`。**永远不要信任前端传过来的 store_id 或 user_id**。

### 3.3 门店线下订单录入 (一致性与防伪)
刚才原型新增了“录入线下订单”功能。
- **数据库事务保障：** 录入时，可能需要建单、可能需要初始化某个暂态客户信息。这种需要多表插入的地方，必须放在单一数据库事务(Transaction)中。
- **幂等性保障 (防抖重提)：** 在网络极差的工棚/新小区环境录入，用户可能会狂点提交。需要前端生成唯一的请求 ID，后端配合 Redis 分布式锁做防重控制，避免同一单子被录入两次。

## 4. 微信小程序生态集成

- **手机号快速获取：** B端人员（老板，工长）入库时，建议后台预先录入他们的手机号。首次登录小程序时，使用 `getPhoneNumber` (需企业认证) 获取手机号，与系统打通直接赋予对应的后台权限，做到“无密登录”。
- **消息订阅与下发：** 利用“订阅消息”能力。比如：针对某个安装工，“有新的工单派发给你了，请点击查看”；或者针对客户，“您的窗户已经完成防水打胶，前往查看”。后端通过 node-weixin-api 调用下发。

## 5. 云托管首选推荐 (运维建议)

针对全栈开发者，强烈建议使用 **微信云托管 (WeRun)** 作为生产环境：
- **免鉴权：** 小程序请求可以无需经过复杂的鉴权/AppSecret维护直接打到云托管内，Node.js 侧可以直接拿到 OpenID。
- **弹性扩缩容：** 后续推广并发大时，能自动拉起更多实例；夜间无人使用可以缩容到 0 收费，省钱省心。
- **CI/CD 集成：** 直接绑定 GitHub/Gitee，push 分支自动发版/构建 Docker 镜像，非常符合前端开发者的自动化工作流。

## 7. 前期准备工作清单 (Checklist)

在正式撸代码前，由于涉及到国内的合规要求以及微信生态，您需要提前准备以下事项（部分流程耗时较长，建议并行推进）：

### 7.1 资质与账号（最耗时，优先级最高）
- **注册企业微信小程序：** 必须使用商业/企业营业执照注册认证。因为系统需要用到 `获取用户手机号` 的 API，个人类型小程序不支持该权限。
- **域名抢注与备案（重点）：** 如果您要在传统的腾讯云轻量应用服务器（CVM）自己搭环境，你**必须**拥有一个已备案的域名，并且配置 HTTPS（SSL证书），微信小程序才能向你的服务器发起接口请求。国内 ICP 备案通常需要 1-2 周时间。（*注：如果采用微信云托管，可暂时免去域名备案烦恼*）。

### 7.2 腾讯云基建准备
- **腾讯云账号认证：** 准备企业或个人实名认证账号。
- **开通对象存储 (COS)：** 用于存放前文提到的海量工地图片资料。提前建好存储桶 (Bucket)，设置跨域规则 (CORS) 允许小程序端直传。
- **购买腾讯云 MySQL 实例：** 建议直接购买云数据库（而不是自己在服务器上用 Docker 搭建），因为云应用会有容灾容灾机制和自动备份，客户订单数据一旦丢失非常麻烦。

### 7.3 开发环境准备
- 全局安装 `Taro CLI` 用于生成前端模板。
- 准备一份基础的 Node.js `Dockerfile`（现在不论是买服务器还是用云托管，容器化部署都是全栈开发的标准实践）。

## 8. 传统服务 vs 微信云托管 及开通指南

您提到打算使用“腾讯云服务器”并询问“云托管”怎么开通，这里为您澄清两者的关系和开通方式：

### 8.1 选传统服务器 还是 微信云托管？
其实**微信云托管的底层资源也是腾讯云**。这两种方案都是大厂云服务，但针对微信开发的体验截然不同：

*   **传统腾讯云服务器（轻量 / CVM）：**
    *   **特点：** 你有完整的服务器 Linux root 权限，可以自己装 Nginx，装宝塔面板等。
    *   **痛点：** 必须自己买域名、等备案、配 SSL 证书；另外前端需要自己拿到 `code` 发给后台，后台去微信服务器按流程换取 `OpenID` 进行登录鉴权。
*   **微信云托管 (WeChat Cloud Hosting) - 极力推荐：**
    *   它专为微信生态打造。对于小团队或个人全栈极为友好。
    *   **爽点 1（免域名免备案）：** 小程序可以直接走微信内网发请求，你不需要去搞域名和备案，立刻就能开发联调测试。
    *   **爽点 2（免登录鉴权）：** 你的 Node.js 后台收到的 HTTP 请求 Header 里，**自动自带**了用户的 `wx_openid`，连换取登录态的代码都省了。
    *   **爽点 3（CI/CD）：** 绑定 Github/Gitee 仓库后，一推代码自动部署打镜像。

### 8.2 微信云托管开通步骤

1. **登录微信公众平台：** 浏览器访问 [mp.weixin.qq.com](https://mp.weixin.qq.com) 并用管理员扫码登录刚刚注册的企业小程序后台。
2. **找到入口：** 左侧菜单栏向下滚动，在 **“开发者工具”** 模块下找到 **“微信云托管”**，点击进入。
3. **环境初始化：** 点击开通，系统会提示你扫码绑定一个 **腾讯云账号**。环境初始化完成后，会默认赠送一个免费的灰度测试环境配额。
4. **数据库与对象存储：** 微信云托管控制台内可以直接一键拉起 MySQL，也可以直接无缝调用云开发的对象存储。
5. **部署 Node 服务：**
   - 进入“服务管理”，新建服务（例如叫 `api-server`）。
   - 部署方式建议选择 **流水线代码部署**，授权你的 Git 仓库。
   - 告诉它你的端口号（比如 Node 跑在 3000），点击发布，你的云端接口就跑起来了。

## 9. 极速开跑：微信云托管行动执行指南 (无需购买传统服务器版)

如果您决定全面拥抱**微信云托管**，抛弃传统服务器买域名、备案的烦恼，请严格按照以下顺序执行。这也是最低初始成本、最快进入写代码状态的路径。

### 🛠 第一阶段：行政与账号准备 (现在开始)
1. **注册企业小程序**：前往微信公众平台，使用**营业执照**注册一个小程序。
   * **需要购买项**：微信认证费 **300元/年**（为了拿到获取用户手机号的权限，这个钱省不掉）。
2. **获取核心密钥**：在小程序后台「开发管理 -> 开发设置」中，拿到 `AppID` 和 `AppSecret`，保存到本地（这在开发中会用到）。

### ☁️ 第二阶段：云端基础设施开通 (完成第一阶段后)
1. **开通云托管**：在小程序后台「开发者工具 -> 微信云托管」点击开通。
   * **需要购买项**：绑定腾讯云账号，并充值几十块钱防欠费（首月通常有免费额度，后续为按量计费，用多少扣多少，极省）。
2. **一键拉起数据库**：在微信云托管控制台，找到左侧的「数据库」模块，新建一个 MySQL 实例。
   * **需要购买项**：按规格购买（选最低配 1核1G 即可，后期随时平滑升级），几十块钱/月，不需要自己运维。
3. **一键拉起对象存储 (图片存这)**：在云托管控制台，开启内置的对象存储。
   * **需要购买项**：按使用量和流量计费，前期图片少，基本只要几块钱。

*(至此，您无需购买任何域名或Linux服务器，云端基建已全部 ready！)*

### 💻 第三阶段：本地写代码的初始化 (今天即可开始！)
既然云端在走流程或已经搞定，本地代码现在就可以建文件夹开干了。

1. **安装本地开发工具：**
   * 下载并安装 **微信开发者工具**（稳定版）。
   * 确保本地电脑已安装 `Node.js` (建议 v18+ 或 v20+)。
2. **前端代码初始化 (Taro)：**
   ```bash
   # 全局安装 Taro 命令行工具
   npm install -g @tarojs/cli
   
   # 初始化前端小程序工程
   taro init sojoy-mini-app
   # 在交互提示中选择：React 框架，保留 TailwindCSS 支持
   ```
3. **后端代码初始化 (Node.js)：**
   ```bash
   # 推荐使用 NestJS 初始化后端工程
   npm i -g @nestjs/cli
   nest new sojoy-backend
   ```
   **关键提示**：在后端工程根目录写好一个 `Dockerfile`（即使是本地开发，云托管发布全靠它）。
4. **打通联调任督二脉：**
   * 后端代码写好后，提交到 Github/Gitee，在云托管后台配置流水线，一键部署上去。
   * 前端小程序代码中直接调用 `wx.cloud.callContainer`，免鉴权、免配域名向刚刚部署的 Node.js 服务发送请求！

**行动建议**：您现在可以立刻打开终端，执行 `npm install -g @tarojs/cli` 和 `npm i -g @nestjs/cli` 把本地前后端两个脚手架跑起来了！

## 10. 核心技术难点与资深开发者“避坑”指南

从事类似 O2O (线上到线下服务) 和施工管理的系统开发，从资深开发者的角度来看，实际落地时往往会遇到以下几个“深水区”。提前规避能省下极大的后续重构成本：

### 10.1 弱网环境下的前端容错（地下室/新小区场景）
- **痛点：** 安装工人经常在刚交付的毛坯房、地下室或偏远别墅区施工，4G/5G 信号极差或断续。如果在这种环境下直接让他上传 9 张工地照片，极大概率会 `timeout` 或中途失败，导致工人暴躁不用。
- **资深解法：**
  - **离线持久化队列：** 提交施工日志时，先把数据写到本地 `wx.setStorageSync`。然后前端跑一个轮询队列去慢慢传。
  - **接口防抖与幂等性：** 弱网下用户绝大概率会狂点“提交”按钮。除了前端做 Button `loading` 态和节流之外，后端接口**必须做幂等**（例如前端传一个随机 UUID 作为 `request_id`，后端使用 Redis 挡住同一请求的二次插入，否则会产生大量的重复脏数据）。

### 10.2 大量图片并发上传导致的内存溢出与限流
- **痛点：** 微信小程序的 `wx.uploadFile` 并发数是有限制的（通常最多只能并发 10 个网络请求），且图片不做处理直接转 Base64 或并发上传会引发小程序崩溃 (OOM) 或网络堵塞。
- **资深解法：**
  - **必须强制本地压缩：** 无论用户手机像素多高，上传前使用 `wx.compressImage` 把尺寸控制在 1080p 以下。
  - **切片与串行队列：** 如果用户一次性选了 9 张照片，前端代码绝不能直接 `Promise.all` 扔给 OSS，必须写一个异步队列，2~3 张一组串行上传，拿到一组 url 后再上传下一组。

### 10.3 “数据隔离”与“水平越权”（极其容易翻车的安全漏洞）
- **痛点：** 既然是多门店系统，A门店的店长如果能调用接口查到B门店的订单和客户电话，这就是严重的客源泄露。最常见的新手错误是前端在接口参数里传了 `{ store_id: 'xxx' }`，后端直接用它去查数据库。
- **资深解法（永远不要信任前端）：**
  - 在 B 端接口鉴权中，用户的 `store_id`、`role` 等敏感范围信息应该在登录时封存在 JWT (或 Redis Session) 中。
  - 后端执行 `GET /orders` 的时候，查询条件应该是 `Where store_id = 用户Token解析出来的store_id`。
  - 遇到老板、管理总台等高频高阶查询，需仔细设计好拦截器(Interceptor/Guard)的通用底层逻辑，一旦漏写拦截器就会导致越权。

### 10.4 小程序“提审”之痛：B/C 混合体容易被拒
- **痛点：** 这是一个混合了 B端（管理/工单）和 C端（官网展示、进度查询）的应用。如果初次提交微信审核时，审核员打开一看全是登录框和内部管理菜单，大概率会以“涉及内部管理系统”被拒。
- **资深解法：**
  - **C端展示前置：** 小程序的首页(pages/index/index) 必须是无需登录就能看见的“内容库”（比如 19分贝门窗的品牌页、案例展示页）。
  - **B端入口隐藏：** B端的入口可以做在“我的”页面，点击特定的入口（比如“员工通道”），才进行角色鉴权并跳转到内部工作台。
  - **提审账号：** 提审时务必提供一个测试账号的路径和密码给微信审核员，明确备注您的应用包含公开案例展示与内部进度流转功能。

### 10.5 微信“订阅消息”的触达幻觉
- **痛点：** 产品想得很好：订单分配了自动给工人发微信，完工了自动给客户发微信。但在当前的微信规则下，这很难做到无缝。因为微信目前的“订阅消息”是**一次性授权**，即：用户在小程序里点一次授权框，你才能发一条消息。
- **资深解法：**
  - 在业务关键节点（例如：工人点击“接受工单”按钮、客户点击“提交量尺预约”按钮）时，把这些按钮封装成触发订阅授权的方法，诱导用户勾选【总是保持以上选择，不再询问】。
  - 不要依赖微信消息作为唯一的工单流转通知机制，这很容易漏。必须在小程序内部做一个“红点提醒/站内信”列表兜底。
  - 针对 B端 内部员工的高频通知，建议辅以“企业微信”绑定或企微机器人的 Webhook 推送，触达率才是 100%。

### 10.6 Taro 分包策略（早做打算）
- **痛点：** 微信小程序主包不能超过 2MB。如果不做分包，项目写到一半就会发现真机预览都发不上去。
- **资深解法：**
  - 建项目初期就规划好：主包只放首页、登录页、各个Tab栏的入口页以及全局组件。
  - 把 C 端的具体订单页作为一个分包，把 B 端的所有工作台视图（店长管理、老板驾驶舱、工人接单看板）作为独立子包（SubPackages）加载。哪怕前期页面少，一开始搭建架子就这样做，能避免后期的重构地狱。
